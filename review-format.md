# 査読プロンプト＆フォーマット

## 査読観点

実装とspec.mdを比較する際、以下の観点から検査してください：

### 1. エンティティ整合性
- 仕様書に記載されているエンティティ・コンポーネント・機能が、実装に存在するか
- 実装に存在するが、仕様書に未記載の項目はないか
- 名前は仕様書と実装で一致しているか

### 2. ロジック矛盾
- 条件分岐の処理順序に矛盾がないか
- フラグ・状態遷移の管理に漏れがないか
- デッドバンド・しきい値の定義が仕様通りか
- タイミング・順序依存性が明記されているか

### 3. 実装漏れ
- TODO・FIXME・NotImplemented コメントが残っていないか
- 仕様書の「実装済み」セクションで未対応のものはないか
- 条件分岐で漏れているケース（edge case）がないか

### 4. バグリスク
- 参照・参照先の誤りはないか（存在しないエンティティへのアクセス）
- テンプレート変数の不正な参照はないか
- NaN、undefined、None の処理漏れはないか
- リソースのリーク（ファイルハンドル、接続など）はないか

## 出力フォーマット（必須・厳守）

```markdown
# REVIEW - YYYY-MM-DD HH:MM
<!-- PROJECT: {project-name} -->
<!-- STATUS: pending -->

## 🔴 要対応
- [ ] #001 説明（フォーマット.m5踢）
- [ ] #002 説明

## ✅ 整合確認
- [ ] #003 説明

## ✅ 整合確認済み
- #004 説明（チェックボックス不要）

## 📝 仕様書への追記推薦
- #005 記載推奨事項
```

# 査読フォーマット（REVIEW.md）

## 出力フォーマット（必須・厳守）

```markdown
# REVIEW - YYYY-MM-DD HH:MM
<!-- PROJECT: {project-name} -->
<!-- SUB: {sub_id} -->           ← サブプロジェクトの場合のみ
<!-- STATUS: pending -->

## 🔴 要対応
- [ ] #001 説明（ファイル名:行番号）
- [ ] #002 説明

## 🟡 要確認
- [ ] #003 説明

## ✅ 整合確認済み
- #004 説明（チェックボックス不要）

## 📝 仕様書への追記推薦
- #005 記載推奨事項
```

**ルール:**
- 項目番号は `#001` から連番で付与する
- `🔴 要対応` / `🟡 要確認` にはチェックボックス `- [ ]` を付ける
- `✅ 整合確認済み` / `📝 追記推薦` にはチェックボックスを付けない
- サブプロジェクトの場合は `<!-- SUB: {sub_id} -->` を3行目に追加する

---

## 査読観点

仕様書（spec.md）と実装（implementation_path 以下）を比較する際の観点：

### 1. エンティティ整合性
- 仕様書記載のエンティティ・コンポーネント・機能が実装に存在するか
- 実装に存在するが仕様書に未記載の項目はないか
- 名前が仕様書と実装で一致しているか

### 2. ロジック矛盾
- 条件分岐の処理順序に矛盾がないか
- フラグ・状態遷移の管理に漏れがないか
- デッドバンド・しきい値の定義が仕様通りか
- タイミング・順序依存性が明記されているか

### 3. 実装漏れ
- TODO / FIXME / NotImplemented コメントが残っていないか
- 仕様書の「実装済み」セクションで未対応のものはないか
- 条件分岐で漏れているエッジケースがないか

### 4. バグリスク
- 存在しないエンティティへの参照がないか
- テンプレート変数の不正な参照がないか
- NaN / undefined / None の処理漏れがないか
- リソースリーク（ファイルハンドル・接続など）がないか

---

## サブプロジェクトの査読

サブプロジェクト（`{sub_id}/spec.md`）を査読する場合：

- 査読結果は `{sub_id}/REVIEW.md` に出力する
- `<!-- SUB: {sub_id} -->` を3行目に必ず入れる
- `implementation_path` はサブプロジェクト固有のパス（.claudeflow.yaml の `sub_projects.{sub_id}.implementation_path`）を使用する
